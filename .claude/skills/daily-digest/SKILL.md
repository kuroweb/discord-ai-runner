# daily-digest

## Goal

当日の主要なAIニュースを日本語で要約し、Discord向けに読みやすいデイリーダイジェストを作成する。

特に、Claude Code、Codex、Gemini、Cursor など主要なエージェントツールの最新動向、機能追加、価格改定、提供形態の変更、重要な反応は優先的に確認する。

## Workflow

1. WebSearch / WebFetch を使って、当日〜直近24時間のAI関連ニュース候補を広めに集める。
2. まず話題検知用ソースで当日の注目トピックを拾い、その後に一次情報や信頼できる報道で裏取りする。
3. 各候補について公開日・更新日・発生日を確認し、何が新しい出来事なのかを一文で言える状態にする。
4. 下の `Freshness gate` を通過した候補だけを残す。通過しない候補は捨てる。
5. 同じ事象の重複記事をまとめ、一次情報（公式発表・企業ブログ）を優先する。
6. 残った候補を重要度順に並べ、3〜7件ではなく「通過した件数だけ」採用する。件数を埋めるために古い話題を足さない。
7. 各項目を「何が起きたか / 何が新しいか / なぜ重要か」で要約する。
8. 各ニュース項目では、詳細説明の直後に対応するソースリンクを置き、説明と出典を離さない。
9. 不確実な情報は推測せず、確認できた事実のみを書く。

## Freshness gate

候補は以下をすべて満たす場合だけ採用する。

- 新しい出来事が当日または直近24時間以内に起きている。
- その「新しさ」を日付つきで説明できる。
- 単なる再共有、再浮上、まとめ直し、感想記事ではない。
- 本文の `ソース: URL` に置ける裏取りソースがある。

以下のものは原則除外する。

- 数日前から知られていた話題を、別メディアが今日まとめ直しただけの記事
- 過去発表の比較記事、解説記事、ベンチマーク再紹介
- X で再度バズっただけの過去ニュース
- 「最近注目」だが具体的な新イベントがないトピック
- 公開日が新しくても、中身が古い発表の焼き直しの記事

## Source policy

### 話題検知用に優先して見るサイト

- X のトレンドや関連ポスト
- はてなブックマークの人気エントリー
- Zenn のトレンド
- Qiita のトレンド

### 話題検知の進め方

毎回、本文を書き始める前に以下を順に確認する。

1. X で AI、LLM、agent、Claude Code、Codex、Gemini、Cursor などの関連話題を確認する。
2. はてなブックマークの人気エントリーから AI・開発ツール系の話題を確認する。
3. Zenn のトレンドから AI、生成AI、LLM、開発支援ツール系の話題を確認する。
4. Qiita のトレンドから AI、生成AI、LLM、開発支援ツール系の話題を確認する。
5. そこで見つけた話題候補を最低 5 件はメモし、その後に一次情報や報道で裏取りする。

この工程を飛ばして、いきなり既知の大手ニュースだけで本文を組み立てない。

### 本文の根拠として優先するソース

- Claude Code、Codex、Gemini、Cursor など対象ツールの公式ドキュメント、公式ブログ、公式アカウント、公式リリースノート
- 公式発表、企業ブログ、プロダクト更新情報、研究機関の公開情報
- 信頼できる主要メディアや業界メディア
- 必要に応じて複数ソースで事実確認できる記事

## Priority topics

- Claude Code、Codex、Gemini、Cursor の新機能、モデル対応、制限変更、料金変更、提供開始・終了
- 主要エージェントツール間の競争や差分が見える発表
- 開発者の実利用に影響するアップデート、障害、仕様変更
- AI研究・プロダクト全般の大きなニュース

### ソースの使い分け

- X、はてなブックマーク、Zenn、Qiita は「何が話題になっているか」を見つける入口として使う。
- それらを本文の唯一の根拠にしない。可能な限り公式発表または信頼できる報道へ辿って確認する。
- トレンド発の話題を採用する場合も、詳細説明の `ソース: URL` には裏取りに使った主要ソースを載せる。
- 公式情報が存在せず、コミュニティ投稿しか確認できない場合は採用基準を厳しくし、未確認情報として扱うか除外する。
- 英語サイトを使う場合も、`ソース: URL` には元記事の URL をそのまま載せる。

### トレンドソースの評価基準

- 同じ話題が複数のトレンドソースで見えている場合は優先度を上げる。
- X だけで盛り上がっていて裏取りできない話題は採用しない。
- はてなブックマーク、Zenn、Qiita で技術者の反応が強い話題は、開発者影響が大きい候補として優先的に確認する。
- 主要エージェントツールに関する更新は、トレンド入りしていなくても毎回能動的に確認する。

## Recency rules

- 原則として当日または直近24時間の新情報のみ採用する。
- 直近24時間以内に公開された記事でも、中身が古い発表の再掲やまとめ直しであれば採用しない。
- 直近24時間を少し超える話題を入れるのは、その期間内に重要な続報、正式発表、価格変更、障害復旧、提供開始など明確な新展開がある場合だけにする。
- X の再浮上、週次まとめ記事、過去記事の再共有だけでは採用しない。
- 発生日と記事公開日がずれている場合は、何が新しいのかを確認できたときだけ採用する。
- 日付が確認できないソースは優先度を下げ、他ソースで日付が確認できなければ除外する。
- 「最新」「今日の動向」として書く場合は、本文中に具体的な日付を入れる。
- JST を基準日付として扱う。UTC の日付境界で判断しない。

## Output format

- 1行目: タイトル用の短い要約（平文・20文字程度）
- 空行
- 本文: Markdown
  - `## 今日のハイライト`
  - 箇条書きでニュース要点
  - `## 詳細`
  - 各ニュースを `### 見出し` + 2〜4行で要約 + 直下に `ソース: URL`
  - 英語サイトの場合も `ソース:` に元URLを書く
  - 要約内に「何が新しいのか」と具体日付を入れる
  - ニュースごとの説明とリンクは必ず隣接させる
  - 記事をまたいだ独立した `## ソース` セクションは作らない

### 詳細セクションの例

```md
## 詳細

### OpenAIが新機能を発表

2026-03-20 に正式発表された新機能の概要。今回新しく何が変わったか。なぜ重要か。
ソース: https://example.com/article
```

## Constraints

- 日本語で書く。
- 誇張や煽り表現は避け、事実ベースで記述する。
- 日付は可能な限り具体的に書く。
- トレンド系サイトの話題性と、本文に使う根拠ソースを混同しない。
- 古い話題を新着ニュースとして混ぜない。新規性が弱いものは落とす。
- 採用件数が少なくてもよい。鮮度を満たさない話題で穴埋めしない。
- トレンドソース確認を省略しない。探索不足のまま本文を書かない。
